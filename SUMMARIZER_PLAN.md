# LiteAgent: Universal Dual-Model Summarizer Architecture

This document outlines the architectural plan to inject an intelligent "Summarizer Node" that applies to **ALL tools**—both your internal python tools (e.g., `read_file`, `run_shell_command`) and external MCP tools (like `explore_codebase`).

## Architecture Overview
By moving the interception point to the **Central Tool Registry** (`src/liteagent/tools/registry.py`), we can compress *any* massive payload generated by any tool.

We will use a fast, lightweight local model (like `qwen2.5:0.5b`) to compress the data into a dense, bulleted summary *before* it returns to the Brain. 

Crucially, **the Brain remains in total control.** We dynamically inject a `will_summarize` parameter into the schema of every allowed tool, allowing the Brain to explicitly opt-out and fetch the raw text if it needs verbatim lines.

---

## 1. Core Configuration (`src/liteagent/core/config.py`)

Add the following fields to your configuration manager:
- `summarizer_enabled: bool = True`
- `summarizer_provider: str = "ollama"` 
- `summarizer_model: str = "qwen2.5:0.5b"`
- `summarizer_threshold: int = 4000` (Minimum character length to trigger compression)
- `summarize_tools: list[str] = ["*"]` (Whitelist of tools to compress. e.g., `["*"]` for all, or `["read_file", "search_in_files", "explore_codebase_tool"]`)

## 2. The Summarizer Service (`src/liteagent/core/summarizer.py`)

Create a dedicated module that dynamically instantiates the correct Provider.

**The Context-Aware Prompt:**
The Prompt is injected with the *original Tool Description*. This forces the tiny model to extract precisely what the Brain expects.

```text
You are an internal data extraction node for an autonomous coding agent.
You have received a massive raw output payload from the tool '{tool_name}'.

The main agent expects this tool to do the following:
"{tool_description}"

Your goal is to extract and compress the raw payload so the main agent gets exactly what it expects.
Do NOT hallucinate. Keep the response as a dense, bulleted summary.

PAYLOAD TO SUMMARIZE:
{payload}
```

## 3. Universal Registry Interception (`src/liteagent/tools/registry.py`)

Instead of hooking into `mcp_manager.py`, we hook directly into the `ToolRegistry` class. When any tool is registered (via `register` or `register_external_tool`), we do two things:

### Step A: Schema Injection
If the tool is in our `summarize_tools` whitelist, we append a new parameter to its JSON schema:
```python
schema["parameters"]["properties"]["will_summarize"] = {
    "type": "boolean",
    "description": "Set to false to bypass the auto-summarizer and get the full raw output. Defaults to true to save context tokens.",
    "default": True
}
```

### Step B: Execution Wrapper
We wrap the actual tool function (`func`) in an asynchronous `summarizing_wrapper` before saving it to `self.tools[name]`.

```python
async def summarizing_wrapper(**kwargs):
    # 1. Pop out the agent's control flag
    will_summarize = kwargs.pop("will_summarize", True)
    
    # 2. Execute the underlying tool natively
    if inspect.iscoroutinefunction(func):
        result = await func(**kwargs)
    else:
        result = func(**kwargs)
        
    result_str = str(result)
    
    # 3. Route to Summarizer if needed
    needs_compression = (
        settings.summarizer_enabled and 
        will_summarize and 
        tool_allowed and 
        len(result_str) > settings.summarizer_threshold
    )
    
    if needs_compression:
        from core.logger import log_event
        log_event("summarizer_triggered", "registry", {
            "tool": tool_name,
            "raw_size": len(result_str),
            "threshold": settings.summarizer_threshold
        })
        
        summary = await summarize_payload(tool_name, tool_desc, result_str)
        
        log_event("summarizer_completed", "registry", {
            "tool": tool_name,
            "summarized_size": len(summary)
        })
        
        return f"[NOTE: Raw payload was {len(result_str)} chars and was auto-summarized]\n\n{summary}"
        
    return result_str
```

## Why this is the Ultimate Architecture
- **Universal Application:** Your internal tools (like `run_shell_command` dumping a massive `npm install` log) get the exact same compression benefits as remote MCP tools.
- **Flawless Agent Control:** The LLM natives knows it can pass `{"will_summarize": false}` anytime it needs verbatim text.
- **Zero Remote Tool Modification:** The remote tools never receive the injected parameter because it is cleanly popped out in the wrapper before execution.

## 4. Observability and Debugging

To ensure you have full visibility into the agent's behavior, we will aggressively use `log_event` and `log_error` throughout the Summarizer service.

**Key Logs to Implement:**
1. `summarizer_schema_injected`: Logged in `registry.py` when a tool's schema is successfully modified to include `will_summarize`. (Includes the tool name).
2. `summarizer_triggered`: Logged before calling the LLM, showing the raw payload size vs the threshold.
3. `summarizer_completed`: Logged after the LLM completes, showing the final compressed size and the time it took.
4. `summarizer_bypassed`: Logged if a tool breached the threshold, but the Brain agent explicitly passed `will_summarize=False`!

This will allow you to tail the agent's session log files and see exactly when and why data is being compressed.
